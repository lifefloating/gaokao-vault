# 学校省份绑定与学校专业补全设计

## 背景

当前项目的 `schools` 表已经存在 `province_id` 字段，但现有学校抓取链路并未稳定写入该字段。`SchoolSpider` 主要依赖 `sch_id=1..5000` 的详情页穷举方式发现学校，能够覆盖大量学校详情页，但不会从院校检索页提取学校所属省份。结果是学校基础信息已经入库，院校所在地省份却长期为空。

同时，`SchoolMajorSpider` 当前只遍历 `schools` 表中的学校，再从学校详情页中的 `major-list` 区域提取 `data-code`，并据此关联 `majors` 表。这个链路有两个天然缺口：第一，只有已发现并入库的学校才会进入学校专业抓取；第二，如果页面上的专业链接缺少 `data-code`，或者 `data-code` 与 `majors.code` 未命中，当前逻辑会直接跳过，导致部分学校专业关系缺失。

用户本次明确的目标是：

1. `schools.province_id` 仅表示学校所在地省份，不表示招生省份。
2. `school_majors` 继续保持全国唯一关系，不按招生省份拆分。
3. 保留现有 `sch_id` 穷举详情页作为兜底发现方式。
4. 新增“按省份筛选的院校分页列表”作为补充发现入口，用来补全学校集合和学校所在地省份。
5. 在不改数据库 schema 的前提下，补强学校专业抓取覆盖率。

## 目标与非目标

### 目标

- 让 `SchoolSpider` 除详情页穷举外，还能从院校检索列表中发现学校。
- 让 `SchoolSpider` 在学校入库时尽可能写入正确的 `province_id`。
- 让 `SchoolMajorSpider` 在 `data-code` 之外支持更稳妥的后备匹配策略。
- 保持现有三阶段编排不变，继续由 `schools` 驱动 `school_majors`。

### 非目标

- 不新增按招生省份维度拆分的学校专业关系表。
- 不修改 `school_majors` 的唯一键。
- 不重构 orchestrator、pipeline 或 dedup 机制。
- 不因为列表页解析失败而移除现有 `sch_id` 穷举抓取逻辑。

## 设计概览

整体方案采用“保留兜底 + 增加发现源 + 强化匹配”的方式：

- `SchoolSpider` 仍然保留对 `sch_id=1..5000` 详情页的穷举抓取。
- `SchoolSpider` 增加一个新的列表发现子流程，从院校检索列表页进入，遍历省份筛选结果与分页结果，从列表卡片中提取学校标识与候选省份。
- 列表页不直接入库 `schools`，而是只负责调度学校详情页，并通过 `meta` 传递候选 `province_id`。
- 学校详情页仍然是 `schools` 的唯一正式入库来源，负责合并详情页字段与列表页候选省份。
- `SchoolMajorSpider` 保留现有按专业代码匹配的主路径，同时增加按专业名精确匹配的后备路径，并对歧义命中进行显式告警而不是错误绑定。

## 学校发现与省份绑定

### 请求流

`SchoolSpider.start_requests()` 将包含三类请求：

1. 现有 warmup 请求，用于建立访问学校相关页面所需的会话状态。
2. 院校检索入口请求，用于启动“按省份筛选的院校分页列表”发现链路。
3. 现有 `sch_id=1..5000` 学校详情请求，作为兜底发现来源。

列表页解析流程负责：

- 识别当前省份筛选上下文。
- 解析当前页中的学校卡片，提取 `sch_id`、学校名、候选省份名。
- 跟进分页链接，直到当前省份的所有分页抓完。
- 为每个列表项调度对应的学校详情页请求。

### 详情页作为唯一入库点

学校详情页仍然是 `SchoolItem` 的唯一产出点。这样做有两个原因：

- 详情页字段最全，避免列表页和详情页分别 upsert 导致内容竞争。
- 可以在一个地方统一处理字段合并、校验、哈希和去重。

列表页只提供候选信息，不直接写库。详情页在构建 `SchoolItem` 时会同时读取：

- 详情页自己解析出的字段。
- 列表页通过 `meta` 传进来的候选 `province_id`。

### 省份解析优先级

Spider 内部增加 `provinces` 维表缓存，并提供名称到 `province_id` 的解析函数。省份字段合并规则固定为：

1. 若详情页能够明确解析出学校所在地省份，则使用详情页值。
2. 若详情页无法解析，而列表页 `meta` 中存在候选 `province_id`，则使用列表页值。
3. 若两者都没有，保持 `province_id = NULL`。

该规则保证详情页为主、列表页为辅，避免来源冲突。

### 调度去重

由于同一所学校可能同时被 `sch_id` 穷举和省份列表页发现，`SchoolSpider` 需要维护进程内去重集合，至少以 `sch_id` 为键，避免重复调度相同详情页请求。这样可以减少重复抓取，并降低一次运行中的无效页面访问。

## 学校入库保护规则

现有 `upsert_school()` 会在冲突更新时直接用 `EXCLUDED.province_id` 覆盖旧值。这对本次需求不安全：如果某次详情抓取没有拿到省份，空值会覆盖已经存在的正确省份。

因此，学校 upsert 需要调整为：

- 当新记录中的 `province_id` 非空时，正常覆盖。
- 当新记录中的 `province_id` 为空时，保留已有记录中的 `province_id`。

这条规则只针对 `province_id`，其余学校字段仍按照现有 upsert 逻辑更新。这样可以避免历史正确省份被脏回写为 `NULL`。

## 学校专业补全

### 当前问题

`SchoolMajorSpider` 当前逻辑只处理“链接上存在 `data-code`，且 `majors.code` 中存在对应记录”的情况。任何缺少 `data-code`、代码不一致、或页面只暴露专业名称的场景都会被静默跳过。

### 新的匹配策略

学校专业关联解析改为两步：

1. 从学校页面尽可能提取原始专业标识，包括：
   - `data-code`
   - 链接地址中的专业参数
   - 专业名称文本
2. 按固定优先级解析 `major_id`：
   - 第一层：按 `code` 精确匹配 `majors.code`
   - 第二层：若没有可用 `code`，但有专业名称，则按 `name` 精确匹配 `majors.name`
   - 第三层：若按名称匹配到多条记录，则判定为歧义，不写入，只记录 warning

这里不做模糊匹配，也不按部分名称猜测，避免把学校专业关系错误地绑定到错误的全国专业记录上。

### 诊断与可观察性

对于无法匹配到 `major_id` 的专业项，需要记录结构化 warning，至少包含：

- `school_id`
- `sch_id`
- 学校页面 URL
- 原始 `data-code`
- 原始专业名称

这能为后续修正规则或补充专业维表提供直接证据。

## 错误处理与回退策略

- 若列表页 DOM 结构变动，列表发现子流程可以部分失效，但 `sch_id=1..5000` 的兜底抓取仍继续工作。
- 若某个分页解析失败，只跳过该页，不让整个 spider 崩溃。
- 若省份名称无法映射到 `provinces` 维表，记录 warning 并保持 `province_id` 为空。
- 若学校专业无法稳定匹配 `major_id`，记录 warning 并跳过该条关系，不做猜测性绑定。

这种策略的核心是：允许补充链路失效，但不允许破坏现有主链路，更不允许错误入库。

## 测试策略

测试需要覆盖以下行为：

1. 学校列表页解析：
   - 能从列表项中提取 `sch_id`
   - 能提取学校名
   - 能提取候选省份或分页链接
2. 学校详情页省份合并：
   - 详情页有省份时，优先使用详情页
   - 详情页无省份但列表 `meta` 有候选值时，使用候选值
   - 候选值为空时不覆盖已有正确值
3. 请求去重：
   - 同一 `sch_id` 被多个来源发现时不会重复调度详情页
4. 学校专业关联：
   - 有 `data-code` 时按 `code` 正常入库
   - 无 `data-code` 但按名称唯一命中时可入库
   - 同名歧义时跳过并告警

同时必须保留现有回归行为：

- `SchoolSpider.parse()` 对 404 的跳过行为不变。
- 现有学校详情字段解析与 tag 解析行为不退化。
- `SchoolMajorSpider` 对正常 `data-code` 页面保持兼容。
- orchestrator 的 Phase 2 / Phase 3 顺序不变。

  - 增加专业原始标识提取、分层匹配、歧义告警

  - 调整 `province_id` 的 upsert 保护逻辑，避免空值覆盖已有非空值
- `tests/`
  - 增加学校列表解析、省份合并、学校专业 fallback 的单元测试

## 验收标准

- 运行 `schools` spider 后，新发现学校可以通过列表页补充进入学校详情抓取。
- 已入库学校在后续抓取中能稳定写入或保留正确的 `province_id`。
- 运行 `school_majors` spider 后，学校专业关系数量相较当前版本增加，且不存在基于歧义名称的错误绑定。
- 现有学校详情解析、Phase 编排、去重逻辑与数据库 schema 保持兼容。
